NFT API Updates
 
We're trying something new! The Alchemy team is hard at work improving the NFT API for you, and we want to be sure you know what we're working on. This is our first newsletter in support of that goal. As a NFT API customer, expect these in your inbox moving forward. Today, we're excited to share some new features that have shipped!

While we continue to focus on serving fast, reliable, and scalable NFT ownership data and NFT metadata (i.e. images, attributes, etc), we are starting to expand NFT API capabilities with new endpoints, as you’ll see below.

We’ve built these because of your asks to date, and we want to continue hearing from you — this will help us to iterate and improve.

Get Started Today:
 
getNFTsForCollection:
  • Based on user feedback, we are opening a new endpoint to allow devs to get all tokens for a collection / contract, e.g., Bored Apes here. Check out the docs! There was complexity in building this endpoint, since neither erc721 or erc1155 standards have a method that returns all the tokens in a contract, so this required a fair amount of indexing and creativity.
           
          Use ENS names in owner field for getNFTs:
          • We have seen that users are eager to utilize their ens name in the owner parameter. So we built this functionality! Here’s an example fetching NFTs for kunalroy.eth.
               
              Handling any token ID format:
              • Based on user demand, we now accept token IDs to the getNFTMetadata api in the form of an integer, a hex string or an extended hex string. The response payload will contain a tokenId in extended hex string format, which is how IDs are encoded on the blockchain. So regardless of the format you send us, you will always access the same metadata.
               
              Flow Contract Filters on getNFTs API:
              • The Flow getNFTs API now supports parameters (contractAddresses[] and contractNames[]) to filter NFTs by contract for an owner in the getNFTs endpoint. Check out the docs!
                   
                  Flow New Contract Support:
                  • We are constantly working on increasing contract coverage support on our Flow NFT API! Last week we added support for 10 new contracts (including Rarible) on Flow Mainnet and 3 new contracts on Flow Testnet. Check out the full list!
                       
                       
                       
                       
                       
                      What Else We’re
                      Working On:
                       
                      We are eager to share some work with you that isn’t quite ready to be tested, but we’re excited about it and think you will be too. We’re hoping to ship the following in the coming weeks:
                       
                      Improving NFT Metadata Hit Rate (the percentage of requests that we received where we sent back valid metadata) — NFTs are all unique in their combination of structural and contractual attributes, and we’re on a mission to index them all for our users!
                       
                      • Cryptopunks and Cryptokitties. Since these collections predate erc721 or erc1155 standards, we are working on special handling for these contracts.
                       
                      • Gaming NFTs such as Decentraland, Sandbox, and NFTWorlds, keep metadata completely off-chain, behind their own API. These require special handling, and we are working to integrate with each one of these (and more).
                       
                      • New OpenZeppelin uri format. The new template from OpenZeppelin contains a new uri format for erc1155 that we will soon handle.
                           
                          • "Reveal" collections. There is a common pattern in NFT launches today where a collection will have valid metadata, but the metadata will change after the "reveal" or "launch". We are devising a strategy to be able to get this revealed metadata as soon as it’s available.
                           
                          • "Provenance" collections. Some collections keep pointers to their metadata completely off-chain. These collections do not even support the tokenUri or uri methods, and therefore do not conform to the erc721 or erc1155 standards. Instead, they have a centralized and undocumented web page that provides a lookup to metadata. One example is HashMasks.
                           
                          Image Caching
                           
                          • Many of you have asked for an image hosted on Alchemy servers that you can display for an NFT. This will allow developers to avoid unknown third party servers and abstract away any formatting or resizing. We are currently working on this and will update you when it is live.
